## Классы сбоев

| Класс | Повторяющийся паттерн | Что теперь должно делать правило |
|---|---|---|
| Premature closeout | Ответ завершался summary, хотя следующий шаг был internal and Codex-eligible. | Продолжать выполнение в текущем task, если route совместим; иначе дать один цельный handoff block. |
| Missing external instruction | Если работа фактически оставалась пользователю, не было четкого action package. | Финал должен содержать `## Инструкция пользователю` или явно сказать `Внешних действий не требуется.` |
| Wrong user boundary | GitHub repo creation был оставлен пользователю при доступном `gh`. | Проверять `gh`/connector; если owner/name ясны и write path доступен, выполнять repo/remote/push внутри Codex. |
| Evidence overclaim | Scenario marked `passed` до ключевого transition result. | Для brownfield no-repo pass requires created repo или explicit blocker. |
| Generated tooling mismatch | Template script не работал после materialization в generated repo. | Verify scripts обязаны поддерживать и template repo, и generated project root. |
| Archive policy gap | Retained source/evidence попадали в active language checks. | Generated repo должен явно разделять active human layer и archival/source exceptions. |

## Уроки для ChatGPT Project

1. Для каждого direct task сначала выполнять repo-first router, затем фиксировать self-handoff по repo rules, и только потом переходить к remediation.
2. Не считать advisory text переключателем профиля или reasoning mode: надежная единица маршрутизации только новый task launch либо текущий repo-confirmed route.
3. Не завершать field step как `passed`, пока нет фактического evidence artifact: repo path, remote URL, commit hash, validator output или explicit blocker.
4. Любое действие, которое Codex может выполнить сам в текущей среде, не должно превращаться в user instruction.
5. Если внешний шаг действительно остается, инструкция должна содержать exact action, path/URL, command if applicable, expected result and verification.
6. Runtime remediation OpenClaw+ нельзя смешивать с repo lifecycle conversion: live runtime changes требуют отдельного stabilization task, rollback plan and before/after acceptance.
