# Пример качественного reality-check

## Текущее состояние проекта
- В репозитории уже есть модуль интеграции, но журналирование ошибок включено только частично.
- В конфигурации подтверждён флаг `retry_enabled=true`, однако лимит повторов не документирован.

## Что сказано в официальной документации
- Документация клиента API требует явного таймаута на каждый исходящий запрос.
- Для повторных попыток рекомендуется exponential backoff, а не фиксированная задержка.

## Какие есть реальные кейсы, issue-треды и известные баги
- В issue трекере есть два кейса, где без явного таймаута поток зависал на сетевом слое.
- На форуме интеграции описан workaround: логировать код ответа и тело ошибки раздельно.

## Где есть расхождения между проектом и внешними источниками
- Проект использует фиксированную задержку 5 секунд, документация рекомендует экспоненциальную схему.
- Проект не фиксирует таймаут на уровне вызова, хотя документация считает это обязательным.

## Предварительный вывод
- Решение нужно строить вокруг явного таймаута, расширенного логирования и замены fixed retry на backoff.
- До кодовых изменений нужно отдельно проверить совместимость текущего retry-механизма с уже развернутой средой.
