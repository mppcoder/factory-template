# Проверка реальности

## Текущее состояние проекта
- [PROJECT] Повторы для email уже реализованы через поле `retry_count` и сервис очередей версии `v3.4`.
- [PROJECT] Для HTTP-интеграции временные ошибки сейчас записываются в лог и завершают процесс без повторной постановки в очередь.
- [PROJECT] Журнал попыток доставки уже хранит `request_id`, `status`, `error_code`, `attempt_number` в `src/integration/delivery_log.py`.

## Официальная документация
- [DOC] Документация внешнего API рекомендует повторять запросы только для сетевых ошибок и ответов `429`/`503`.
- [DOC] Документация рекомендует использовать идемпотентный ключ для повторных попыток и сохранять его между ретраями.
- [DOC] В release note `v3.4` зафиксировано требование журналировать все попытки retry.

## Реальные кейсы и известные баги
- [REAL] В issue команды интеграций подтверждено, что кратковременные ошибки `503` возникают несколько раз в неделю.
- [REAL] В обсуждениях постморема указано, что отсутствие retry уже приводило к потере части уведомлений.
- [FIX] В email-канале ранее уже внедряли ограниченный retry с безопасным ограничением по числу попыток.

## Сопоставление и расхождения
- Проект уже умеет считать попытки, но не применяет это к HTTP-каналу.
- Документация подтверждает допустимость retry для временных ошибок, а фактическая реализация этого не делает.
- Реальные кейсы подтверждают полезность изменения и отсутствие retry как источник потерь.

## Вывод
Изменение совместимо с текущей архитектурой: нужен ограниченный retry на три попытки, сохранение идемпотентного ключа и обновление журналирования без изменения внешнего контракта API.
