# Пример хорошего заполнения: инвентаризация системы

## 1. Общая картина системы
- Название системы: Внутренний сервис синхронизации заявок
- Основное назначение: собирает заявки из внешней формы и передает их в CRM.
- Текущее состояние: частично работает
- Где расположена основная среда: один VPS и отдельная managed DB

## 2. Репозиторий и кодовая база
- Основной repo: company/request-sync
- Дополнительные repo: нет
- Основные ветки: main, dev
- Есть ли расхождение между main/dev/prod: prod запущен не с последнего commit main
- Что в repo уже выглядит устаревшим или заброшенным: старая папка `legacy-scripts/`

## 3. Окружения и точки запуска
- Prod: systemd service на VPS
- Stage / test: нет отдельного окружения
- Локальный запуск: через docker compose
- Кто и как обычно запускает систему: разработчик вручную через compose и на сервере через systemctl

## 4. Компоненты системы
- Компонент 1: API-приемник заявок
- Компонент 2: очередь обработки
- Компонент 3: модуль отправки в CRM
- Что из этого подтверждено как активное: API и CRM-модуль
- Что предположительно уже не используется: старый экспорт в CSV

## 5. Данные и хранилища
- Базы данных: PostgreSQL
- Файловые хранилища: локальная папка для логов
- Кэши / очереди: Redis
- Есть ли миграции и насколько они актуальны: есть, но последние две не применялись на prod

## 6. Конфиги, секреты и переменные среды
- Главные конфиги: `.env`, `docker-compose.yml`, `config/production.yaml`
- Где лежат переменные среды: в `.env` локально и в systemd EnvironmentFile на сервере
- Есть ли расхождения между repo и боевым окружением: да, на prod есть дополнительные переменные
- Какие конфиги выглядят критичными: CRM API credentials и DB URL

## 7. Внешние интеграции
- API / внешние сервисы: CRM API, почтовый сервис
- Авторизация / SSO: нет
- Почта / мессенджеры / уведомления: SMTP
- Платежи / CRM / storage / аналитика: CRM и базовая веб-аналитика

## 8. Что уже подтверждено как работающее
- Работает 1: прием входящих webhook
- Работает 2: запись в базу
- Работает 3: базовая отправка в CRM

## 9. Что частично работает или вызывает сомнения
- Под вопросом 1: повторная обработка после временной ошибки CRM
- Под вопросом 2: ротация логов
- На чем основано сомнение: разные показания из кода и production-логов

## 10. Что не найдено или не подтверждено
- Не подтверждено 1: актуальный backup policy
- Не подтверждено 2: реально ли используется локальный stage
- Что нужно проверить отдельно: cron jobs и состояние Redis persistence

## 11. Главные выводы inventory
- Главный вывод 1: система живая, но не совпадает с repo полностью.
- Главный вывод 2: перед изменениями нужно синхронизировать картину по конфигам и миграциям.
- Что это означает для следующего этапа: сначала repo audit и gap register, потом change spec.
