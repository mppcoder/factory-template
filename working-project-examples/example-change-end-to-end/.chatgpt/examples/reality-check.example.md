# Пример хорошо заполненной проверки реальности

## Текущее состояние проекта
- В шаблоне уже существует `.chatgpt/` как слой нормализации.
- Есть `scenario-pack/`, `stage-state.yaml`, `task-index.yaml`, `policy-presets.yaml` и `project-presets.yaml`.
- Пустые шаблоны ключевых артефактов содержат минимум структуры, но не дают достаточно наглядного образца глубины заполнения.

## Что сказано в официальной документации
- Внутренний стандарт фабрики требует, чтобы handoff в Codex шел только по нормализованным артефактам.
- Внутренний стандарт также требует держать core универсальным и не делать brownfield-ядро под один проект.

## Какие есть реальные кейсы, issue-треды и известные баги
- На предыдущих итерациях пользователю не хватало примеров хорошо заполненных файлов, из-за чего шаблоны были понятны по структуре, но не по глубине.
- Отдельно был выявлен риск, что английские формулировки внутри файлов ухудшают восприятие и создают неоднозначность.

## Где есть расхождения между проектом и внешними источниками
- Структура уже достаточно зрелая, но содержимое отдельных шаблонов пока беднее, чем ожидает практическое использование.
- Формально шаблон поддерживает pipeline, но человеку всё ещё неочевидно, как должен выглядеть хороший `user-spec` или `reality-check`.

## Предварительный вывод
- Решение нужно строить не через смену core-архитектуры, а через усиление содержимого шаблонов.
- Можно безопасно добавить папку примеров и русские подсказки по заполнению без нарушения текущих скриптов и валидаторов.
