# Project Knowledge / знания проекта

Этот каталог нужен для долгоживущих знаний о проекте.

Здесь храним не разовые заметки, а информацию, которая помогает через недели и месяцы:
- что за проект и зачем он существует;
- как устроена архитектура;
- как безопасно выкатывать и поддерживать поставку;
- какие ограничения и устойчивые правила уже подтверждены.

## Когда обновлять

Обновляйте этот слой, когда:
- принято устойчивое продуктовое или архитектурное решение;
- изменился подтвержденный deployment-процесс;
- выявлены новые ограничения, которые будут влиять на будущие задачи.

Не обновляйте этот слой по мелочам вроде временных экспериментов.

## Структура

- `project.md` — миссия, контекст, границы и целевая ценность проекта.
- `architecture.md` — ключевые архитектурные решения, компоненты и trade-offs.
- `deployment.md` — как доставлять изменения и что проверять перед выкладкой.
- `constraints.md`, `patterns.md`, `known-issues.md`, `glossary.md`, `fixes-registry.md` — вспомогательные устойчивые знания.

## Связь со спецификациями

Рабочие документы feature живут в `work/features/<feature>/`:
- `specs/user-spec.md` отвечает на вопрос "что хочет пользователь";
- `specs/tech-spec.md` отвечает на вопрос "как агент это реализует";
- `decisions.md` хранит важные решения, отклонения и проверки после выполнения задач.

В `project-knowledge/` переносите только устойчивые выводы из `decisions.md`, а не каждый рабочий шаг.

## Правило качества

Если новый участник команды не может по этим файлам понять:
- что вы строите,
- почему именно так,
- и как это безопасно выпускать,

значит слой нужно доработать.
